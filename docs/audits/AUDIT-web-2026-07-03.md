### P1 — Major

**[P1] Bottom sheets have no focus trap, dialog role, or Escape-to-close**
- **Location:** `apps/web/src/features/map/ui/PlaceModal.tsx:31`, `apps/web/src/features/flag/ui/FlagSheet.tsx:44`
- **Category:** Accessibility
- **Impact:** Keyboard and screen-reader users can tab past the sheet into the map/background content while it's open, and there's no way to dismiss it without a mouse. `AppHeader`'s dropdown (same file, lines 13–31) already implements both click-outside and Escape handling — these two sheets are the outliers.
- **WCAG:** 2.1.2 No Keyboard Trap, 2.4.3 Focus Order, 4.1.2 Name/Role/Value (missing `role="dialog"` / `aria-modal`)
- **Recommendation:** Extract a shared `<BottomSheet>` primitive that sets `role="dialog"` + `aria-modal="true"`, traps focus, restores focus on close, and closes on Escape — then use it in both places.
- **Suggested command:** `/impeccable harden`
- **Status (2026-07-03 fix pass): RESOLVED.** Added `shared/ui/BottomSheet.tsx` — `role="dialog"` + `aria-modal="true"`, Tab-trapped focus, Escape-to-close, focus restored to the trigger on close, plus a `shared/model/dialog-stack.ts` so the nested case (opening `FlagSheet` from inside `PlaceModal`) only traps/closes the topmost sheet. Both components now use it; both z-index values also moved from raw `z-10`/`z-20` to the semantic `z-modal` token as part of the same edit.
- **Correction (2026-07-04):** the `z-modal` claim above was wrong. Tailwind v4's `@theme` block silently dropped the `--z-*` custom properties (unrecognized namespace), so `z-modal` compiled to nothing and both sheets were actually rendering at `z-index: auto` this whole time. The dialog semantics (role/focus-trap/Escape) were genuinely correct and unaffected; only the z-index half of this status was false. Now actually fixed — see the z-index finding below.

**[P1] Live GPS coordinates drive the discoveries query key, causing redundant fetches**
- **Location:** `apps/web/src/features/seek/ui/SeekPage.tsx:18-25`, `apps/web/src/features/seek/api/discovery.queries.ts:4-7`, `apps/web/src/features/map/model/use-geolocation.ts:30-44`
- **Category:** Performance
- **Impact:** `useGeolocation` uses `watchPosition` with `enableHighAccuracy: true` and no throttling, so `coords` updates on every GPS tick. `SeekPage` passes that raw `lat`/`lng` straight into `useNearbyDiscoveries`'s query key, so every micro-jitter in position is a new TanStack Query cache key — triggering a fresh network request. This is the opposite of what `docs/PERFORMANCE.md` and the CLAUDE.md "would it feel fast on a Moto G over 3G?" litmus test call for: it burns data and battery on exactly the low-end/patchy-connection users this product targets.
- **Recommendation:** Round coordinates to a stable precision (e.g. ~3 decimal places, ~110m) before building the query key, or only re-query when the user has moved past some meaningful distance threshold from the last-queried center.
- **Suggested command:** `/impeccable optimize`
- **Status (2026-07-03 fix pass): RESOLVED.** Both halves of the recommendation landed: `discovery.queries.ts` now rounds lat/lng to 3 decimals (~110m) before building the query key, so GPS jitter no longer mints a new cache key/fetch; and `use-geolocation.ts` now applies a 20m haversine distance filter (the CLLocationManager `distanceFilter` pattern) before calling `setState` at all, so sub-threshold GPS noise no longer re-renders `SeekPage`/`MapView` either. Raw coordinates still drive the live map center and user-location marker for movement past the filter.

**[P1] Session and in-progress report data are memory-only** *(new, from critique)*
- **Location:** `apps/web/src/app/store/index.ts:19` (only `{theme, mapRadius}` is persisted), `apps/web/src/features/auth/model/session.slice.ts` (accessToken/sessionUser), `apps/web/src/features/report/ui/ReportPage.tsx` (draft form state)
- **Category:** Performance / Robustness
- **Impact:** On the low-end Android / patchy-connectivity devices this product explicitly targets, tabs get reloaded under memory pressure routinely. Because the session token and the in-progress report draft are held only in memory, a reload silently logs the user out and — since `/report` is a `ProtectedRoute` — discards whatever they had typed, with zero explanation. This directly undermines the same "fast & calm on patchy connectivity" pillar the GPS query-key finding above already flags.
- **Recommendation:** Persist the access/refresh token and the in-progress report draft to `localStorage`/`sessionStorage` alongside the existing `theme`/`mapRadius` persistence.
- **Suggested command:** `/impeccable harden`
- **Status (2026-07-04 fix pass): RESOLVED.** `store/index.ts`'s `partialize` now also persists `accessToken` and `sessionUser` — a reload no longer logs the user out. New `features/report/model/report-draft.slice.ts` moves `ReportPage`'s form state (product/place/price/quantity) from local `useState` into a persisted Zustand slice, cleared only on successful submit; the wizard `step` itself stays local/ephemeral (a reload mid-draft returns to the "form" step with fields already filled, not stuck mid-"confirm" — a natural recovery point). Also added: `shared/api/http.ts` now clears the session on any `401` response, since persisting the token makes a stale/expired/rejected token a real recurring case instead of a rare one — without this, a reload with an invalid persisted token would leave the user "authenticated" with every request silently failing rather than being redirected to sign in. Verified end-to-end with Playwright: filled the report form, confirmed the draft in `localStorage`, reloaded, confirmed both the authenticated state and the exact form values survived (the first verification attempt gave a false failure from a test-script mistake — `addInitScript` re-firing on the reload and overwriting the app's own write — corrected and re-verified).

**[P2] Semantic z-index tokens compiled to nothing — `@theme` silently drops unrecognized custom-property namespaces** *(new, found 2026-07-04 while building dark mode)*
- **Location:** `apps/web/src/app/index.css` (`--z-dropdown`/`--z-sticky`/etc. were declared inside `@theme`), consumed via bare class names at `apps/web/src/shared/ui/ToastViewport.tsx:55` (`z-toast`), `apps/web/src/shared/ui/OfflineBanner.tsx:11` (`z-sticky`), `apps/web/src/shared/ui/BottomSheet.tsx:72` (`z-modal`)
- **Category:** Theming / Correctness
- **Impact:** Tailwind v4's `@theme` block only auto-generates utilities (and even preserves the custom property at all) for its own recognized token namespaces (`--color-*`, `--font-*`, `--radius-*`, `--shadow-*`, `--animate-*`, …). `--z-*` is not one of them — declaring it inside `@theme` causes Tailwind to silently drop the custom property from the compiled output entirely, and no `.z-toast`/`.z-sticky`/`.z-modal` utility class is generated either. Every element using these class names has been rendering with `z-index: auto` since they were introduced (`ToastViewport`/`OfflineBanner` predate this session; `BottomSheet` was added in an earlier pass in this same fix series and was never actually applying a z-index despite the status line above claiming otherwise). Confirmed via `getComputedStyle` (`zIndex: "auto"`) and empirically: a new floating element placed at a location that happened to overlap `SeekPage`'s location-denied banner had its clicks intercepted by the banner underneath it, because the new element had no real z-index either.
- **Recommendation:** Declare the z-index scale as plain `:root` custom properties (not inside `@theme`), and reference them in class names with Tailwind v4's CSS-variable shorthand: `z-(--z-modal)` (equivalent to, and preferred over, `z-[var(--z-modal)]`).
- **Suggested command:** `/impeccable harden`
- **Status (2026-07-04 fix pass): RESOLVED.** Moved the z-scale to a plain `:root { --z-dropdown: 20; ... }` block in `index.css` (confirmed present in the compiled CSS output). Updated all four consuming components (`ToastViewport`, `OfflineBanner`, `BottomSheet`, and the new `ThemeToggle`) to `z-(--z-name)`. Verified via `getComputedStyle` (now resolves to the real number) and via an end-to-end Playwright click test (previously failed with a pointer-interception timeout, now succeeds). `AppHeader`/`SeekPage`/`ProductPicker` still use raw numeric `z-10`/`z-20`/`z-50` — those aren't broken (Tailwind's built-in numeric scale works natively), just not yet migrated to the semantic names; tracked in the P2 finding above.
- **Status (2026-07-04, later): FULLY RESOLVED.** `AppHeader.tsx`, `MapView.tsx`, `SeekPage.tsx`, and `ProductPicker.tsx`'s dropdown all now use `z-(--z-sticky)`/`z-(--z-dropdown)` — zero raw numeric z-index classes remain anywhere in `apps/web/src`. Confirmed via `grep`; `tsc --noEmit`/`eslint`/full Jest suite all clean.

**[P2] Form validation errors aren't associated with their fields or announced** *(new, found 2026-07-04 during `/impeccable audit` re-pass)*
- **Location:** `apps/web/src/features/auth/ui/SignInPage.tsx:74,89`, `SignUpPage.tsx:98,190,207,224`, `apps/web/src/features/report/ui/PriceInput.tsx:50`, `apps/web/src/features/report/ui/ReportPage.tsx:123,129` — 9 instances total. (The higher-level submit-failure banners — `SignInPage:93`, `SignUpPage:103/149/230`, `FlagSheet:74`, `ConfirmStep:33` — already correctly use `role="alert"`; this finding is specifically about the per-field errors.)
- **Category:** Accessibility
- **Impact:** Every per-field validation error (`{errors.email && <p className="text-error ...">}`-style) renders as plain text with no `role="alert"`, no `aria-describedby` linking it to its `<input>`, and no `aria-invalid` on the input itself. A sighted user sees the red text appear; a screen-reader user gets no signal an error appeared at all, and even if they find it, has no programmatic link back to which field it belongs to. `grep -rn "aria-describedby\|aria-invalid"` across `apps/web/src` returns zero matches — this is a systemic gap, not a one-off. React Hook Form's default `shouldFocusError` does move focus to the first invalid field on `SignInPage`/`SignUpPage`'s submit, which partially mitigates those two files; `ReportPage`'s plain-`useState` validation has no such focus behavior at all, so it's the more severely affected of the two patterns.
- **WCAG:** 3.3.1 Error Identification, 4.1.2 Name/Role/Value, 4.1.3 Status Messages

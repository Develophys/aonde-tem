## Overall Impression

The coordinate-integrity fix is the best piece of work across all five rounds: a shared predicate used identically by the validation gate and the UI's own confirmation affordance, with a comment warning future editors not to let the two drift apart, and traced end-to-end to confirm there's genuinely no path around it. That's the standard the rest of the app should be held to. Two new findings this round are worth real attention, though smaller in blast radius than last round's bug: a silent price-corruption path in `PriceInput` (typing "." instead of "," silently produces a 100x-wrong price with no error), and a real performance issue in the map layer (every search/radius change fully tears down and rebuilds the map's marker layers instead of just updating data, working against the project's stated performance pillar). Neither is as severe as last round's phantom-coordinates bug, but both are the same *class* of problem — a silent, unvalidated path through a core data-entry or rendering flow.

## Priority Issues

**[P2, new] `PriceInput` silently strips "." and can produce a 100x-wrong price with no error shown.**
Confirmed by trace: `PriceInput.tsx:15` strips any character that isn't a digit or comma, so typing "12.50" (a "." decimal, common on many phone numeric keypads) becomes "1250" with the "." simply discarded — no comma, no decimal point. `parseFloat("1250")` = 1250, which passes the `> 0 && <= 99_999.99` bound (line 25), so `onChange(1250)` fires with no validation error at any point. A user intending to report R$12,50 can silently submit R$1250,00, discoverable only if they read the ConfirmStep recap carefully — directly against PRODUCT.md's "glance, not read" model for the one fact (price) that must be glanceable-*correct*.
**Fix**: normalize a typed "." to "," instead of stripping it (treat both as the decimal separator), so no input silently loses its intended meaning.
**Suggested command**: `/impeccable harden apps/web/src/features/report/ui/PriceInput.tsx`

**[P2, new] Map marker layer fully rebuilds on every discoveries update instead of updating data in place.**
Traced via the effect's dependency array: `DiscoveryMarkerLayer.tsx:162` includes `discoveries`, so every search/radius/refetch change re-runs the effect — and the cleanup function (lines 152-161) unconditionally removes both layers and the source *before* the new effect body runs. That means `applyLayers()`'s fast-path (`if (map.getSource("places")) { setData(...); return; }`) never actually fires on updates, because the source was just torn down; every update takes the full rebuild path (recreate source, recreate both layers, re-register all click/hover handlers) instead of a single cheap `setData()` call. This works directly against `docs/PERFORMANCE.md`'s "performance is a pillar" mandate on low-end Android, especially since `discoveries` can change on every debounced keystroke in `SeekPage`'s search.
**Fix**: split into two effects — one that creates the source/layers/handlers once (dependent only on `mapRef`), and a second, `discoveries`-dependent effect that only calls `setData()` on the existing source.
**Suggested command**: `/impeccable optimize apps/web/src/features/map/ui/DiscoveryMarkerLayer.tsx`

**[P3, carried over — open for a second round] FlagSheet's radiogroup still lacks the keyboard behavior its ARIA roles imply.**
Unchanged from round 4: `role="radiogroup"`/`role="radio"` on independently-tabbable buttons, no roving tabindex, no arrow-key navigation. Screen-reader/keyboard users get 5 tab stops instead of the expected single-stop-then-arrow-keys pattern, at exactly the moment (reporting harmful content) where a confusing interaction is costliest.
**Fix**: use `ProductPicker`'s keyboard handling as the template — either drop the ARIA roles if these stay plain toggle buttons, or implement the real radiogroup pattern.
**Suggested command**: `/impeccable polish apps/web/src/features/flag/ui/FlagSheet.tsx`

**[P3, new, minor] `AppHeader.tsx` hardcodes a duplicate of the `--header-inset-top` token instead of referencing it.**
Line 71 recomputes `calc(env(safe-area-inset-top, 0px) + 12px)` inline, one branch away from line 53 which already references the token — exactly the kind of token drift `index.css`'s own comment on that token warns against.
**Fix**: use `var(--header-inset-top)` at both call sites.
**Suggested command**: `/impeccable polish apps/web/src/features/auth/ui/AppHeader.tsx`

## Persona Red Flags

**Ana (reporting a price on the store floor, glancing not reading)** — the persona most exposed to the new PriceInput finding: if her keyboard offers "." instead of ",", her reported price is silently wrong by 100x with nothing to catch it before or after typing.

**Diego (screen-reader user, reporting inappropriate content)** — still hits the same FlagSheet gap flagged last round: Tab visits 5 separate reason buttons instead of the single-stop, arrow-key radiogroup pattern the roles claim.

**A frequent searcher scrolling/typing through many queries** — is the one most exposed to the map-rebuild performance issue: every keystroke-driven search update triggers a full marker-layer teardown/rebuild rather than a cheap data update, on a low-end Android device this is exactly the persona `docs/PERFORMANCE.md` is written for.

## Minor Observations

- `DiscoveryMarkerLayer.tsx:99` hardcodes `"#ffffff"` for the marker's circle-stroke-color rather than importing it from `MAP_COLORS`, inconsistent with that module's own stated purpose as "the one permitted set of duplicated hex literals."
- `ToastViewport.tsx` nests `role="alert"`/`role="status"` rows inside a parent `aria-live="polite"` container — `role="alert"` is itself an implicit assertive live region, so some screen readers may double-announce error toasts.
- `PlacePicker.tsx`: if a user taps a nearby-place suggestion, then hand-edits the resulting text afterward, `hasRealCoords` still passes but the retained coordinates are the original suggestion's, not the edited name's — a low-severity name/coordinate mismatch the checkmark won't catch, since it only validates coordinate presence, not name/coordinate correspondence.

The distance fix took a genuinely different, well-reasoned approach (client-side haversine, since the schema doesn't support per-item distance) rather than forcing the original suggested fix — good judgment, even though it left a new small gap. ## What's Working

1. **Progressive disclosure in `ReportPage.tsx`** — gating Price/Quantity behind Product+Place being filled is a real, well-reasoned improvement (with an in-code comment explaining the one-way reveal and reload-safety), even though it stops short of true sequential steps.
2. **`SeekPage.tsx`'s clearance-band fix** — `left-4 right-20` with an explicit inline comment tying the value back to the recenter/report FAB geometry. This is the second time this exact z-tier/DOM-order bug pattern has been fixed, and both times it's been generalized with a documented reason, not silently patched.
3. **`PlaceModal.tsx`'s distance fix** — recognizing that the per-item schema doesn't carry distance (every item shares one place) and computing it once at the right level, instead of mechanically forcing distance into each item row as originally suggested. Good judgment over literal compliance.

## Priority Issues

**[P1] Report flow's step counter still doesn't match its structure.**
`ReportPage.tsx` gates fields behind progressive disclosure now, but it's still one scrolling screen with a "1 de 2 / 2 de 2" label implying discrete, navigable steps that don't exist. This is the same tension flagged in rounds 1 and 2, now narrower in scope (the crowding is better) but still unresolved at its core.
**Fix**: pick a direction — either split into real sequential screens (Produto → Local → Preço/Quantidade → Confirmar, "N de 3" or "N de 4"), or drop the step-counter framing and present it honestly as one form with a progress hint.
**Suggested command**: `/impeccable shape apps/web/src/features/report`

**[P2] ProductPicker's dropdown still has no blur/outside-click dismissal.**
A touch user (no keyboard, no Escape) who taps away from the product field is left with the results list open, potentially obscuring the PlacePicker field directly below it on a small screen. Unchanged since round 2.
**Fix**: add a document-level pointerdown/blur listener, matching the pattern already proven in `AppHeader.tsx`'s own dropdown.
**Suggested command**: `/impeccable polish apps/web/src/features/report/ui/ProductPicker.tsx`

**[P2] BottomSheet still has no backdrop; `--z-modal-backdrop` remains dead code.**
Sheets float directly over live map/page content with no dimming, and nested sheets (FlagSheet over PlaceModal) have no visual cue for which is "on top" beyond the focus trap. Unchanged since round 2.
**Fix**: add a scrim div at `--z-modal-backdrop` inside `BottomSheet`, with click-to-close — or, if a backdrop-less sheet is a deliberate performance/perceptual choice for low-end devices, remove the unused token and note the decision in DESIGN.md so it reads as intentional rather than incomplete.
**Suggested command**: `/impeccable polish apps/web/src/shared/ui/BottomSheet.tsx`

**[P3] Distance silently disappears when location is unavailable.**
`PlaceModal.tsx`'s new distance display goes from present to fully absent (not degraded) when `userCoords` is null — exactly the denied/loading state `SeekPage` otherwise handles with visible fallback copy elsewhere in the same app.
**Fix**: render a neutral placeholder ("Distância indisponível") instead of omitting the segment entirely, for parity with how location-denied is handled elsewhere.
**Suggested command**: `/impeccable clarify apps/web/src/features/map/ui/PlaceModal.tsx`

**Alex (in-store, wants a fast report filed)** — still pays the cost of the report flow not being true steps: filling product+place+price+quantity on one continuously-growing screen while standing in an aisle is slower than a real 2-4 screen wizard would be, especially one-handed and rushed.

**Riley (community trust-builder)** — benefits from the Aging Amber fix: the freshness legend now reads as a single, trustworthy signal system again, not diluted by reuse as a generic warning color elsewhere in the app.

## Minor Observations

- `ErrorBoundary.tsx` uses `console.error` directly for its catch — fine for now (CLAUDE.md's Logger-port rule is scoped to backend/domain code), but worth wiring to real telemetry eventually.
- `SearchBar.tsx`'s clear ("×") button uses `p-3` + `text-xl leading-none` rather than an explicit `min-h-11 min-w-11` like other icon-only controls in the app (`ToastViewport.tsx`, `PlaceModal.tsx`'s close button) — likely lands close to 44px in practice, but worth an explicit size class for consistency and to stop relying on font-metric math.
- `PlacePicker.tsx`'s "✓ selected place" confirmation reuses `text-fresh` (green) — consistent with the Trust Green Rule (green = confirmed/act on it), not a violation, just worth noting so it isn't mistaken for a freshness label out of context.

## Questions to Consider

- Is BottomSheet's missing backdrop actually a deliberate perceptual/performance choice for low-end Android (one less paint, no backdrop-filter cost), or an oversight — worth deciding explicitly either way rather than leaving `--z-modal-backdrop` as dead code that looks unfinished?
- Now that two rounds in a row have fixed a real issue while a comparable one appeared elsewhere, is it worth a dedicated pass auditing every `absolute`/`fixed` floating element's z-tier and bounding box against every other one, rather than finding these one at a time?

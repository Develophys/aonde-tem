## Priority Issues

**[P1] Manually-typed place names silently default to `lat: 0, lng: 0`, and nothing catches it before submit.**
Confirmed directly: `PlacePicker.tsx:78` — `const base = value ?? { lat: 0, lng: 0 };` runs whenever a user types into the place-name field without first tapping a nearby-place suggestion or "Usar minha localização atual" (the exact fallback path the UI itself recommends when location is denied, `PlacePicker.tsx:101-102`). `ReportPage.tsx`'s `validateForm()` only checks `!form.place` truthiness, never coordinate validity, so the report submits successfully. The "✓ {name}" confirmation (`PlacePicker.tsx:108-110`) renders identically whether the coordinates are real or phantom, so nothing about the UI signals a problem — the user proceeds through Confirm and the celebratory success screen believing they've helped, while the Report may render nowhere sensible (or literally in the Gulf of Guinea) for everyone else. This directly undermines the "map-first," community-trust core of the product.
**Fix**: gate submission on real coordinates — either disable/error on `validateForm` when `place.lat === 0 && place.lng === 0`, with copy directing the user to pick a suggestion or use current location, or don't allow free-text place entry to carry coordinates at all without an explicit selection step.
**Suggested command**: `/impeccable harden apps/web/src/features/report/ui/PlacePicker.tsx`

**[P2] Map clusters are not clickable — a dead tap target on the core screen.**
Confirmed: `DiscoveryMarkerLayer.tsx` registers `map.on("click", "places-points", ...)` (line 104) but never a corresponding handler for the `places-clusters` layer it also creates (lines 78-88). Tapping a cluster circle currently does nothing at all — no zoom, no feedback — which reads as "broken app" rather than "no result here," on exactly the map-first screen the whole product is built around.
**Fix**: add a click handler on `places-clusters` that queries the cluster's expansion zoom (MapLibre's `source.getClusterExpansionZoom`) and flies the map there, matching the near-universal "tap cluster to zoom in" convention.
**Suggested command**: `/impeccable layout apps/web/src/features/map/ui/DiscoveryMarkerLayer.tsx`

**[P3] Bare "✓" Unicode glyph breaks the hand-authored icon vocabulary.**
`PlacePicker.tsx:109` uses a literal "✓" character where every other confirmation moment in the app (EmptyState, ErrorBoundary, ReportPage's success screen) uses a hand-drawn SVG stroke icon.
**Fix**: swap for the same stroke-checkmark SVG used in ReportPage's success screen.
**Suggested command**: `/impeccable polish apps/web/src/features/report/ui/PlacePicker.tsx`

**[P3] FlagSheet's reason list claims ARIA radiogroup semantics without the expected keyboard behavior.**
`FlagSheet.tsx`'s reason options use `role="radiogroup"`/`role="radio"`, but each option is an independently-tabbable `<button>` rather than a single Tab stop with arrow-key navigation between options — the contract those ARIA roles imply. Screen-reader/keyboard users get 5 Tab stops instead of the expected roving-tabindex pattern, at exactly the moment (reporting harmful content) where a confusing interaction is costliest.
**Fix**: either switch to native `<input type="radio">` styled to match, or implement roving tabindex + arrow-key handling.
**Suggested command**: `/impeccable polish apps/web/src/features/flag/ui/FlagSheet.tsx`

## Persona Red Flags

**A frequent community reporter with weak signal indoors** — is the persona most exposed to the P1 finding. `PlacePicker.tsx`'s own copy actively routes this exact user ("Localização negada — informe o local manualmente") into the silent `lat:0, lng:0` path. Their contribution appears to succeed — draft persists, success screen celebrates — while quietly corrupting the shared map data other users rely on. For a product whose entire value is community-sourced location data, this is the worst-case failure mode: confident, invisible, and undermines trust in every other Report once discovered.

**A shopper scanning the map for the cheapest nearby option** — hits the dead cluster tap target the moment results are dense enough to cluster, which is precisely when they most need to drill in. A tap with zero feedback reads as "the app is broken," not "nothing here."

**A keyboard/screen-reader user reporting inappropriate content** — encounters FlagSheet's non-standard radio-group keyboard behavior at exactly the moment a confusing interaction is most likely to cause them to abandon or mis-select a reason.

## Minor Observations

- `PlaceModal.tsx`'s "Ver no mapa" footer button only renders when `data?.coords` is present; low risk given current data shape, but there's no visible fallback dismiss-to-map if that's ever falsy for a place with items.
- `SignInPage.tsx` disables the submit button while pending but not the input fields — a double-submit via Enter-then-click race is theoretically possible, though low-impact given how fast the mutation resolves.
- Autocomplete in `ProductPicker`/`PlacePicker` shows no loading indicator distinguishing "still debouncing" from "no results yet" from "genuinely no results" — minor, since both are fast in practice.

## Questions to Consider

- Given the checkmark in PlacePicker currently confirms *any* typed name regardless of coordinate validity, how much of the report flow's "confidence" UI (checkmarks, the confirm step, the success screen) is actually load-bearing versus decorative — and would surfacing the coordinate problem break that confidence illusion in a good way?
- Should free-text place entry be allowed to carry coordinates at all, or should "Usar minha localização atual" / picking a suggestion be the only paths that can produce a submittable place — trading a little flexibility for eliminating a whole class of silent data corruption?
- Now that a static, unresponsive cluster marker exists on the core map screen, is a visibly "tappable" cluster (a distinct badge/affordance) worse for trust than not clustering at all, if the tap doesn't do anything yet?

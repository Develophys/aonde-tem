## Overall Impression

Real, careful follow-through: four of five issues were fixed correctly and generally (tokens and shared components, not one-off patches), and the GhostButton consolidation even documents its own rationale in-code. But the pattern that caused the original P0 — a transparent, full-width absolutely-positioned element sharing a z-tier with a smaller floating control, DOM-order deciding who wins — has resurfaced at a second location (the map's recenter FAB), which the team hasn't seen yet. And the deepest issue from the first pass, the report flow's structure, wasn't touched. The mechanical fixes are done; the two pieces of real design work (the report flow, and a durable fix for "floating map chrome" collisions) are still ahead.

## What's Working

1. **`GhostButton` fix** (`shared/ui/GhostButton.tsx`) — the component's own doc-comment names the three prior divergent implementations it replaces, and it's actually adopted everywhere it needed to be. A model example of fixing drift by removing the ability to drift, not just patching each site.
2. **`--header-clearance` token** (`index.css`) — the P0 fix was generalized into a reusable token rather than a one-off `top-4` → `top-20` patch, and it's already reused by `ReportPage.tsx`'s header bar too.
3. **`ToastViewport`'s pause/resume** — tracks *remaining* time across multiple pause cycles rather than naively resetting on each hover, a detail most "fix the timer" patches skip.

## Priority Issues

**[P1] Report flow's four fields are still simultaneous — the deepest issue from the last pass, unaddressed.**
`ReportPage.tsx` still renders ProductPicker, PlacePicker, PriceInput, and QuantityStepper on one screen; a "1 de 2 / 2 de 2" counter was added without splitting the screen, so the UI now promises a paced flow it doesn't structurally deliver. Fails the same 3 cognitive-load checklist items as before (single focus, one-thing-at-a-time, minimal-choices-at-once) on the app's core conversion action.
**Fix**: either finish what the step counter implies (split into real sequential per-decision screens) or remove the counter if staying single-screen is the intentional call.
**Suggested command**: `/impeccable shape apps/web/src/features/report`

**[P1] New — the recenter FAB can be blocked by SeekPage's own transparent overlay elements.**
`MapView.tsx`'s recenter button sits at `bottom-24 right-4` (44×44, `z-(--z-sticky)`). `SeekPage.tsx`'s fetch-error card, EmptyState wrapper, and freshness-legend container are all `absolute left-0 right-0` (full viewport width) at `bottom-20`, same z-tier, and render later in the DOM — the exact "same z-tier, later DOM wins" pattern that caused the original P0. Their vertical bands overlap the FAB's, and a plain `<div>` intercepts pointer events across its full bounding box regardless of visible content, so taps on the recenter button can be swallowed by these wrapper elements whenever a fetch error, empty result, or the freshness legend is showing.
**Fix**: constrain these wrappers' width to exclude the FAB's column (e.g. `right-20` instead of `right-0`), or set `pointer-events-none` on the transparent shell and re-enable it only on the visible inner content.
**Suggested command**: `/impeccable layout apps/web/src/features/seek/ui/SeekPage.tsx`

**[P2] Aging Amber repurposed as a generic warning color, against the system's own "One Accent Rule."**
Confirmed in `OfflineBanner.tsx:11` (`bg-aging`) and `SeekPage.tsx:60` (`text-aging`) — both non-freshness contexts. DESIGN.md is explicit that Aging Amber applies "only ever... to freshness labels... not a second and third accent to decorate with."
**Fix**: use `text-text-muted`/`border-border` for these, or deliberately add a real neutral-warning token if one's needed.
**Suggested command**: `/impeccable colorize apps/web/src/shared/ui/OfflineBanner.tsx apps/web/src/features/seek/ui/SeekPage.tsx`

**[P2] Distance data exists but isn't surfaced where PRODUCT.md says it must be glanceable.**
Correcting the initial scan here: `distanceMeters` **is** part of the discovery API contract (`packages/contracts/src/discovery.ts:24`) — it's not missing from the data model. But `PlaceModal.tsx`'s item rows show product name, price, quantity, freshness, and an optional note — never distance, even though it's already on the wire. PRODUCT.md names "distance/where" as one of the three facts that must be legible at a glance, alongside price and freshness; today a user has to eyeball the map instead of reading it in the sheet.
**Fix**: render `item.distanceMeters` (formatted, e.g. "350m") alongside price/freshness in the item row — this is a display-layer gap, not a data or API change.
**Suggested command**: `/impeccable layout apps/web/src/features/map/ui/PlaceModal.tsx`

**[P3] `ProductPicker` dropdown doesn't close on blur or outside click; `BottomSheet` has no backdrop.**
Confirmed: `ProductPicker.tsx` only closes its results list on Escape or selection (no `onBlur`), so tabbing/tapping past it into the adjacent PlacePicker field leaves it open, floating over that field. Separately, `BottomSheet.tsx` renders no scrim/backdrop element at all (`--z-modal-backdrop` is declared in `index.css` but never referenced anywhere in the codebase), so sheets have no dimming and no click-outside-to-close.
**Fix**: close `ProductPicker` on blur (with a short delay so it doesn't race the `onMouseDown` selection, matching the pattern `AppHeader.tsx` already uses for its own dropdown); either wire a backdrop into `BottomSheet` at `--z-modal-backdrop`, or remove the unused token if sheets are meant to stay backdrop-less by design.
**Suggested command**: `/impeccable polish apps/web/src/features/report/ui/ProductPicker.tsx apps/web/src/shared/ui/BottomSheet.tsx`

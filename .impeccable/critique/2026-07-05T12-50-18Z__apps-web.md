## Anti-Patterns Verdict

**Does this look AI-generated? No — but it has drifted from its own rules.**

**LLM assessment**: The app does not read as generic AI slop. No purple/indigo gradients, no glassmorphism, no neon-on-black, a single disciplined accent hue, the system font stack, mostly-flat surfaces, restrained shadow use exactly per DESIGN.md's Elevation rules. This is genuinely careful, deliberate work — not vibe-coded. The failure mode here isn't "looks like AI," it's small, real deviations from the design system the project already wrote down for itself:
- A third border-radius value (`rounded-lg`, 8px) appears once, where the system's Two-Radius Rule permits only `rounded-control` (12px) and `rounded-full`.
- A bare `✓` glyph appears next to raw coordinates in the report flow, where every other confirmation moment in the app uses a hand-authored SVG stroke checkmark — a small break in icon vocabulary consistency.
- Three ghost-style buttons ("Denunciar," "Editar," "Cancelar") each render at a different text size than the system's documented Label-size (14px) spec for ghost buttons — none matches, and they don't even agree with each other.

**Deterministic scan** (`detect.mjs` over `apps/web/src`): 1 advisory finding — a literal `rgba(0,0,0,0.35)` box-shadow color on the user-location map marker (`map/ui/MapView.tsx:60`) that sits outside the documented DESIGN.md palette/tonal ramps. Low severity (a shadow color, not a UI-color token), and plausibly fine as scoped, one-off use — but it's technically undocumented and worth either adding to the system or explicitly ignoring. No other findings; the codebase is clean by the detector's stricter, mechanical pass. This is a much shorter list than the LLM review's drift findings above, because the detector only catches literal-value violations, not typography-scale or radius-token drift expressed via existing utility classes.

**Visual overlays**: Not available in this environment — no browser automation tool is configured, so the live-server/injection overlay step was skipped per the critique flow's fallback rule. This report is based on a full source-code read (JSX/TSX structure, Tailwind classes, and the CSS token file), not a rendered screenshot. If you want the browser-based overlay pass, it needs Playwright/Puppeteer or similar wired into this environment.

## Overall Impression

This is a disciplined, well-considered product surface — better than most first critiques see. The team clearly wrote DESIGN.md's rules *before* drift set in, and most of the app honors them: the accent color stays scoped to its one job, the shared BottomSheet primitive is reused everywhere a sheet-over-map pattern is needed, and there's real accessibility engineering (the ProductPicker combobox, in particular, is correct ARIA, not boilerplate). The single biggest opportunity is closing the gap between the system as documented and the system as shipped — a handful of small, mechanical fixes (radius, ghost-button sizing, toast target size) plus one real layout bug (the header/search-bar overlap) would take this from "good" to "excellent." The report flow's cognitive load is the one structural (not cosmetic) issue worth real design time.

## What's Working

1. **Fetch-error vs. empty-result distinction** (`features/seek/ui/SeekPage.tsx`) — most apps collapse "the network failed" and "genuinely nothing here" into one generic empty state. This app deliberately doesn't, which directly serves the "patchy connectivity" persona called out in the product brief. It's a small decision most teams skip.
2. **`ProductPicker`'s combobox** (`features/report/ui/ProductPicker.tsx`) — a fully correct ARIA combobox/listbox pattern (`role="combobox"`, `aria-activedescendant`, roving highlight, full arrow/enter/escape keyboard handling). Real accessibility engineering, not a checkbox-ticking gesture.
3. **`BottomSheet` as the one signature component** (`shared/ui/BottomSheet.tsx`) — correctly handles nested sheet stacking (flagging content from inside a place-detail sheet), focus-trapping, and focus restoration on close. Exactly the "reuse the primitive, don't hand-roll another panel" discipline DESIGN.md asks for, and it holds up in the one case (nested sheets) that would normally break a naive implementation.

## Priority Issues

**[P0] Search bar can visually bury the sign-in/theme-toggle corner buttons on the home screen.**
- **Why it matters**: `SeekPage.tsx`'s search bar wrapper is positioned `absolute top-4 left-4 right-4` (full width minus 16px margins) with an opaque `bg-surface` background, at the same `z-(--z-sticky)` tier as the `AppHeader` sign-in pill (`fixed right-3`) and `ThemeToggle` (`fixed left-3`), both roughly 44px tall starting ~12px from the top. Because the search bar renders later in the DOM at an equal z-index, on any viewport without a large notch inset (i.e., most Android phones and any desktop browser) it paints directly over both corner controls — on the single most important screen in the app, potentially hiding the only way to sign in.
- **Fix**: Reserve a header band (push the search bar down below the corner buttons' height + safe-area inset) or give `AppHeader`/`ThemeToggle` a strictly higher z-tier than the page content's sticky elements.
- **Suggested command**: `/impeccable layout apps/web/src/features/seek/ui/SeekPage.tsx`

**[P1] The report form puts four decisions on one screen, then a confirm step — not the sequential flow the product intends.**
- **Why it matters**: `ReportPage.tsx`'s "form" step renders ProductPicker, PlacePicker, PriceInput, and a quantity field simultaneously — a 2-step flow (form + confirm), not a staged one. This fails 3 of the 8 cognitive-load checklist items (single focus, one-thing-at-a-time, minimal-choices-at-once) on the app's core conversion action, and is inconsistent with `SignUpPage.tsx`'s correctly-staged email → code → profile flow in the same codebase.
- **Fix**: Split into real sequential steps (place → product → price/quantity → confirm), one primary decision per screen, matching the pattern already proven elsewhere in this app.
- **Suggested command**: `/impeccable shape apps/web/src/features/report` (re-plan the flow), then `/impeccable layout` to implement

**[P1] Ghost-button text size drifts from the documented spec in three different places.**
- **Why it matters**: DESIGN.md specifies Label-size (14px, `text-sm`) for ghost buttons. `PlaceModal.tsx`'s "Denunciar" uses `text-xs` (12px, too small); `ConfirmStep.tsx`'s "Editar" and `FlagSheet.tsx`'s "Cancelar" specify no size class at all and inherit Body (16px, too large). None matches the spec, and the two omissions plus one explicit override mean there's no single accidental pattern to fix in bulk.
- **Fix**: Add explicit `text-sm` to all three, ideally by introducing a shared `GhostButton` component so this can't drift again.
- **Suggested command**: `/impeccable polish apps/web/src/features/map/ui/PlaceModal.tsx apps/web/src/features/report/ui/ConfirmStep.tsx apps/web/src/features/flag/ui/FlagSheet.tsx`

**[P2] Toast dismiss target is under the 44px minimum and auto-dismisses with no pause on hover/focus.**
- **Why it matters**: `ToastViewport.tsx`'s "×" dismiss button is `min-h-6 min-w-6` (24×24px), well under the system's own 44px touch-target rule, and every toast — including error toasts — self-dismisses after a fixed 5000ms with no way to pause it. For a one-handed, distracted, mobile-first user, this is exactly the persona most likely to lose an error toast before they can read or dismiss it.
- **Fix**: Pad the hit area to 44×44px (visual glyph can stay small) and pause the auto-dismiss timer on hover/focus.
- **Suggested command**: `/impeccable polish apps/web/src/shared/ui/ToastViewport.tsx`

**[P2] A stray third border-radius value breaks the Two-Radius Rule.**
- **Why it matters**: `SeekPage.tsx`'s "location denied" banner uses `rounded-lg` (8px), which is neither of the system's two sanctioned radii (`rounded-control` 12px, `rounded-full`). Small, but it's exactly the kind of unenforced drift the system's own rule exists to prevent, and it's the only occurrence in the codebase.
- **Fix**: Change to `rounded-control`.
- **Suggested command**: `/impeccable polish apps/web/src/features/seek/ui/SeekPage.tsx`

## Persona Red Flags

**Jordan (First-Timer)**
- May be unable to find "Entrar" at all on first load, due to the P0 header/search-bar overlap.
- The freshness-color legend only renders once search results exist — a first search that returns zero results never shows it, so the first colored map dot Jordan ever sees later has no on-screen explanation of what the color means.
- The dense, all-at-once report form may read as overwhelming for someone unsure what's required before they start.

**Sam (Accessibility-Dependent, screen reader + keyboard-only)**
- SeekPage's search-error text has no `aria-live`/`role="alert"`, unlike nearly every other error state in the app — a screen-reader user searching on a bad connection may not learn the search failed.
- The radius slider (`<input type="range">`) has an `aria-label` but no `aria-valuetext`, so a screen reader announces a bare number rather than "3.5 km."
- Map markers/clusters are canvas-rendered with only pointer `click` handlers — there's no keyboard path to open a place's details without a mouse or touch.
- The undersized (24px) toast dismiss target compounds against Sam specifically, on top of the general 44px-target violation.

**Casey (Distracted, one-handed, mobile)**
- Directly hit by the P0 overlap bug — the account/sign-in button they expect in the corner may be invisible under the search bar.
- Report-form field errors use negative-margin positioning to tuck under fields, which is fragile if content wraps differently on a smaller device — an error could visually detach from the field it belongs to.
- The 5-second toast auto-dismiss plus 24px close target is a poor combination for someone reading one-handed while walking.

## Minor Observations

- `MapView.tsx`'s user-location marker uses an undocumented `rgba(0,0,0,0.35)` shadow color (flagged by the detector) — low-severity, but either fold it into DESIGN.md's tokens or add it to the ignore list intentionally.
- `DiscoveryMarkerLayer.tsx` tears down and rebuilds the entire MapLibre source/layers on every `discoveries` change (i.e., on every debounced keystroke) rather than updating data in place — risk of visible marker flicker and extra GPU work while typing, which works against the project's stated performance pillar for low-end Android.
- `index.css`'s comment documents the z-index syntax as `z-[var(--z-name)]`, but the codebase consistently and correctly uses the real `z-(--z-name)` shorthand — a stale/misleading comment worth fixing.
- The "Denunciar" (flag/report) action in `PlaceModal.tsx` is a low-contrast, `text-xs` caption-styled link at the bottom of a dense row — undersells a trust/moderation action, given the product brief's stated priority that trust signals be first-class, not an afterthought.
- No "community confirmation" signal appears anywhere reviewed — the product brief's priority #3 (freshness *and* community confirmation as trust signals) is only half-realized; freshness is well-built, confirmation isn't visible yet.

## Questions to Consider

- What if the report flow were actually split into the sequential steps the product's own information architecture implies — would the lower cognitive load beat the single-scroll form's raw speed, given "post a Report fast" is a stated priority?
- What if the account pill and search bar are overlapping because this was never checked on an actual ~360px-wide phone viewport — what would it take to make that check routine going forward?
- Does a first-time user ever actually learn what the green/amber/gray freshness colors mean *before* they need to trust one, given the legend only appears after there are results to show?
